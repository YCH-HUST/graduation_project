基于YOLO与大语言模型的
中医智能辅助诊疗系统开发与研究
摘要：目的：针对传统中医舌诊在临床应用中依赖医师经验、主观性强、难以实现标准化与客观化分析的问题，探索将人工智能技术引入中医舌象分析与辨证辅助诊疗过程中的可行性，构建一种融合舌象视觉信息与问诊文本语义信息的中医智能辅助诊疗系统，以提升中医诊疗的客观性与辅助决策能力。
方法：整合多源舌象图像数据，构建标准化舌象数据库，并基于改进的YOLO目标检测模型实现舌体区域自动定位及舌苔、舌色等关键舌象特征的检测与量化提取；同时，引入大语言模型对患者主诉与问诊文本进行语义解析，生成结构化症状特征表示。在此基础上，融合舌象视觉特征与症状语义特征，构建中医辨证推理模型，实现证候识别与诊疗建议生成，并通过后置大语言模型输出条目化、可追溯的诊断依据说明，以增强系统结果的可解释性。
结果：实验结果表明，基于YOLO的舌象特征检测模型能够较为准确地完成舌体区域定位及舌象特征识别，多模态特征融合方法能够有效提升中医证候辅助识别的合理性与稳定性。系统整体运行稳定，能够完成从数据采集、智能分析到结果展示的完整流程，具有较好的实用性。
结论：本文构建了一种融合舌象视觉特征与问诊文本语义信息的中医智能辅助诊疗系统，在舌诊客观化、多模态信息融合及诊疗结果可解释性方面具有一定优势。以肺系疾病为应用场景，系统基于改进的YOLO模型实现舌象特征自动检测，并结合大语言模型完成症状语义解析与辨证推理，可生成诊疗建议及可追溯的诊断依据说明。研究结果表明，该系统能够提升中医辨证辅助效率，为中医智能化诊疗提供了一种可行的技术路径与实践参考。
关键词：中医舌诊；YOLO模型；计算机视觉
 
Development and Research of an Intelligent Auxiliary Diagnosis System for Traditional Chinese Medicine Based on YOLO and Large Language Models
Abstract: Objective: Traditional Chinese medicine (TCM) tongue diagnosis relies heavily on physicians’ clinical experience in practical applications, resulting in strong subjectivity and difficulties in achieving standardized and objective analysis. This study aims to explore the feasibility of introducing artificial intelligence technologies into tongue image analysis and syndrome differentiation–assisted diagnosis in TCM, and to construct an intelligent auxiliary diagnostic system that integrates tongue visual information with semantic information from inquiry texts, thereby improving the objectivity of TCM diagnosis and supporting clinical decision-making.
Methods: Multi-source tongue image data were integrated to construct a standardized tongue image database. An improved YOLO-based object detection model was employed for automatic tongue region localization and quantitative extraction of key tongue features, such as tongue coating and tongue color. Meanwhile, a large language model was introduced to perform semantic analysis of patients’ chief complaints and inquiry texts, generating structured symptom representations. On this basis, visual tongue features and semantic symptom features were fused to build a TCM syndrome differentiation inference model, enabling syndrome identification and diagnostic recommendation generation, with a post-processing large language model providing structured and traceable diagnostic explanations to enhance result interpretability.
Results: Experimental results indicate that the YOLO-based tongue feature detection model can accurately accomplish tongue region localization and tongue feature recognition. The multimodal feature fusion approach effectively improves the rationality and stability of assisted TCM syndrome identification. The overall system operates stably and is capable of completing a full workflow from data acquisition and intelligent analysis to result presentation, demonstrating good practicality.
Conclusion: The proposed intelligent auxiliary diagnostic system integrates tongue visual features with semantic information from inquiry texts, demonstrating advantages in the objectification of tongue diagnosis, multimodal information fusion, and result interpretability. Focusing on pulmonary-related diseases, the system combines an improved YOLO model with a large language model to enable automated tongue feature analysis, syndrome differentiation, and generation of interpretable diagnostic recommendations. The results indicate that the system effectively improves the efficiency of TCM-assisted diagnosis and provides a feasible technical approach for intelligent diagnosis in traditional Chinese medicine.
Key words: Tongue diagnosis in TCM; YOLO model; Computer Vision
 
引 言
中医药作为我国传统医学的重要组成部分，在疾病预防、诊断与治疗过程中积累了丰富的临床经验。其中，“望、闻、问、切”四诊合参是中医诊疗的核心方法，而舌诊作为“望诊”的重要内容，通过观察舌质颜色、舌苔变化，能够反映人体脏腑功能状态和疾病发展情况，具有信息量大、获取便捷等特点。然而，在实际临床应用中，舌诊结果高度依赖医师个人经验和主观判断，不同医师之间存在较大差异，难以实现标准化、客观化和规模化应用，这在一定程度上制约了中医诊疗的现代化发展。
随着人工智能技术的快速发展，计算机视觉与自然语言处理在医学辅助诊疗领域得到了广泛关注。近年来，国内外学者开始尝试将图像处理、深度学习等方法应用于中医舌象分析研究。在舌象图像处理方面，早期研究多采用传统图像处理方法对舌体区域进行分割和特征提取，但在复杂背景、光照变化及个体差异较大的情况下，鲁棒性较弱。随着深度学习技术的发展，卷积神经网络被引入舌象分析领域，一定程度上提升了舌体检测与舌象特征识别的准确性。其中，基于YOLO等目标检测模型的研究在舌体区域定位和特征检测方面表现出较好的实时性和检测效果，逐渐成为研究热点。
与此同时，在中医诊疗过程中，“问诊”作为获取患者主观症状信息的重要手段，通常以自然语言形式存在。近年来，大语言模型在自然语言理解与语义分析方面取得了显著进展，为中医问诊文本的结构化处理与语义挖掘提供了新的技术手段。多数研究仍集中于单一模态数据，缺乏对舌象图像信息与问诊文本信息的联合建模与综合利用。
总体来看，现有研究在中医舌象智能分析和问诊文本处理方面已取得一定成果，但仍存在以下不足：一是舌诊分析多停留在单一视觉特征层面，对舌象信息的综合表达能力有限；二是舌象视觉信息与问诊文本语义信息之间缺乏有效的融合机制，难以全面反映中医辨证过程；三是部分智能诊疗模型缺乏可解释性，难以满足中医临床实践中对诊疗依据可追溯性的要求。因此，如何融合多模态信息，构建兼具准确性与可解释性的中医智能辅助诊疗系统，仍是值得深入研究的问题。
基于上述背景，本文围绕中医舌诊智能化与多模态信息融合这一研究方向，提出并实现了一种基于YOLO与大语言模型的中医智能辅助诊疗系统。本文从系统实现的角度出发，综合考虑数据采集、模型构建、特征融合与结果展示等关键环节，采用改进的YOLO目标检测模型实现舌体区域的自动定位及舌苔、舌质等特征的检测与量化提取；同时，引入大语言模型对问诊文本进行语义解析，生成结构化症状特征表示。在此基础上，融合舌象视觉特征与症状语义特征，构建中医辨证推理与辅助决策模块，并通过系统化设计实现完整的诊疗辅助流程。
1 绪论
1.1 研究背景
中医药在我国医疗体系中具有重要地位，其独特的理论体系和诊疗方法在疾病防治与健康管理中发挥着不可替代的作用。中医诊疗强调整体观念和辨证论治，通过“望、闻、问、切”四诊合参获取人体健康状态信息。其中，舌诊作为“望诊”的重要组成部分，通过观察舌体形态、舌质颜色及舌苔变化等特征，对机体脏腑功能、气血运行及病理状态进行综合判断，具有信息直观、获取便捷等优势，在临床实践中被广泛应用。
然而，在传统中医临床实践中，舌诊结果高度依赖医师的个人经验与主观判断，不同医师对同一舌象可能给出不同的诊断结论，缺乏统一的量化标准与客观评价体系。这种主观性不仅影响诊疗结果的一致性，也在一定程度上制约了中医诊疗的规范化、标准化和智能化发展。因此，如何对舌象信息进行客观、稳定和可重复的分析，成为中医现代化研究中的重要问题之一。
随着人工智能技术的快速发展，计算机视觉和深度学习方法在医学图像分析领域取得了显著进展，为中医舌象的客观化分析提供了新的技术手段。近年来，研究者开始尝试将图像处理、卷积神经网络以及目标检测模型应用于舌体区域定位与舌象特征提取，相较于传统图像处理方法，深度学习模型在复杂背景、光照变化以及个体差异条件下表现出更强的鲁棒性。其中，以YOLO为代表的目标检测模型在检测精度与实时性方面具有较好平衡，适用于舌体区域的快速定位及舌象特征检测任务。
另一方面，中医诊疗过程不仅依赖舌象等客观视觉信息，还高度依赖患者主诉和问诊信息。问诊内容通常以自然语言形式存在，包含大量与症状、体征及病情变化相关的重要信息。传统中医智能研究多侧重于单一模态数据分析，忽视了问诊文本中蕴含的丰富语义信息。近年来，大语言模型在自然语言理解与语义建模方面取得突破性进展，为中医问诊文本的结构化解析与语义挖掘提供了新的可能。
在此背景下，将舌象图像信息与问诊文本语义信息进行融合，构建多模态中医智能辅助诊疗系统，成为当前研究的重要发展方向。通过融合计算机视觉与自然语言处理技术，有望在提升舌诊分析客观性的同时，更全面地模拟中医辨证思维过程，为中医诊疗提供可靠的辅助决策支持。因此，开展基于YOLO与大语言模型的中医智能辅助诊疗系统研究，具有重要的理论意义和现实价值。
1.2 国内外研究现状
1.2.1 国外研究现状
近年来，国外在知识图谱（Knowledge Graph, KG）、深度学习目标检测（YOLO系列）与大型语言模型（Large Language Models, LLMs）三类技术上均取得显著进展，并开始尝试将这些技术与医学与中医领域结合以解决知识组织、临床决策支持和多模态诊断问题。
在深度学习与视觉方向，YOLO系列在医学图像快速检测与识别任务上一直是高效检测的主流选择，研究者已将轻量化或任务定制的YOLO模型用于中医相关的图像识别（如舌象、药材切片、药材图像识别等），以实现现场快速鉴别与特征提取[1][2]。针对中医舌诊、舌体特征检测的YOLO改进工作已在近年文献中出现。
大型语言模型及其在医学知识编码上的定制研究成为近期热点。国外与国际期刊报道了将预训练LLM通过领域适配（fine-tuning、prompt engineering、知识注入）用于医学推理与问答的研究。针对中医领域，也出现了以中医知识体系为核心进行微调或知识注入的LLM（“Lingdan”项目），显示出在中医临床推理、处方建议与问答等任务上具有潜力[3]。
Yige Liu等人选取YOLOv5、集成SE注意力机制的YOLOv5（YOLOv5_SE）、YOLOv8以及集成SE注意力机制的YOLOv8（YOLOv8_SE）对慢性胃炎患者舌象进行特征提取，发现YOLOv8模型在舌象特征分类任务中的准确率最高[4]。Chunge Li等人基于Yolo深度学习技术，采用分类方法构建舌像多任务学习模型，实现了中医舌诊中舌色、苔色、裂纹和齿痕的同时识别[5]。Sibo Wei等人提出了一个专门针对中医的LLM，采用两阶段训练流程，注入领域特定知识，通过有针对性的刺激对其进行对齐，增强了诊断和辨证能力[6]。
1.2.2 国内研究现状
在国内，知识图谱在中医药领域的研究起步较快、数量与应用均显著增长：研究主要集中于中医方剂、症候-证型关系、处方-药物（药材）组成、诊疗路径等结构化建模，以及构建可用于临床辅助、教学与问答的中医知识库[7]。中医领域的关键难点在于术语标准不统一、知识融合与层次化建模困难、以及推理方法与可解释性不足等。
在技术实现上，国内研究者广泛采用基于规则与统计/深度学习混合的实体识别（NER）、关系抽取（RE）、图数据库（Neo4j）进行存储与查询，许多工作以中医古籍、方剂辞典、临床病例、药典等为数据来源，构建领域专用的知识图谱并开展问答或临床决策支持系统开发。目前有研究基于Neo4j对中医方剂或肺癌中医诊疗知识进行建库与可视化分析[8]。
国内在视觉模型方面也取得显著进展。将YOLO、ResNet、Transformer等模型用于中药材识别、舌象分析、脉象信号分类等任务，并逐步将视觉特征与知识图谱或大语言模型（LLM）结合，以提升诊断准确性和知识检索能力[9]。近年来“中医+AI”成为国家重点科研方向之一，相关论文数量持续上升，显示出较强的学术与应用潜力[10]。
陈健等人研究发现，基于YOLOv8l构建的结直肠息肉语义分割模型具有良好地预测性能，能够自动对息肉进行定位和分类，并精确描述息肉的像素级轮廓[11]。刘奕清等人整理数据，设计了带权无向图网络，建立了智能体辨证推理模型，利用智能体协作架构实现中医大语言模型与辨证推理模型交互[12]。
1.3 存在问题分析
结合中医舌诊及智能辅助诊疗相关研究现状，当前研究与应用过程中主要存在以下几个方面的问题：
1.3.1 舌诊分析主观性较强，缺乏统一量化标准
在传统中医临床实践中，舌象判断主要依赖医师个人经验，不同医师对舌体形态、舌质颜色及舌苔状态的理解和描述存在差异，缺乏统一的客观量化标准。这种主观性导致舌诊结果一致性和可重复性较差，制约了舌诊结果的数据化处理与智能分析。
1.3.2 舌象图像分析对复杂实际环境适应能力不足
部分研究采用传统图像处理方法或简单深度学习模型进行舌象分析，在光照变化、背景复杂以及个体差异较大的实际采集环境中，模型鲁棒性有限。现有部分方法在舌体区域精确定位及多特征联合分析方面仍有不足，难以满足实际应用需求。
1.3.3 问诊文本语义信息利用不充分
中医诊疗过程中，患者主诉和问诊记录以自然语言形式存在，包含大量重要症状信息。然而，现有中医智能研究多侧重于舌象图像等视觉信息，对问诊文本的语义理解与信息挖掘不足，未能充分发挥问诊信息在中医辨证中的辅助作用。
1.3.4 多模态信息融合机制有待完善
中医诊疗强调多信息来源的综合分析，但当前研究中，舌象图像与问诊文本往往被独立处理，缺乏有效的跨模态特征融合方法，难以全面反映中医辨证论治的整体思维过程，从而影响智能辅助诊疗系统的综合判断能力。
1.3.5 智能诊疗模型可解释性不足
尽管深度学习模型在识别精度方面具有优势，但其内部决策过程往往难以直观解释，无法向医师清晰展示诊断依据和推理过程。这在强调辨证逻辑和诊疗依据合理性的中医临床应用中，限制了智能辅助诊疗系统的可信度与推广应用。
1.4 研究目的和意义
1.4.1 研究目的
本研究以中医肺系疾病为切入点，围绕“望诊（舌象）+问诊（文本）”的跨模态证候识别与推荐，构建一个“YOLO舌象检测—LLM前置语义解析—中置推理模型—LLM后置解释生成”三端一体化应用的智能辅助诊疗系统，并在真实使用情景中验证可用性与可靠性。
1.4.2 研究意义
本研究以肺病场景为牵引，围绕“数据标准—视觉检测—语义解析—推理预测—解释生成—系统闭环”的路线，聚焦当前中医AI在数据质量、跨模态融合、可解释性与临床落地上的关键痛点，具有明确的学术价值与现实意义。
1.5 研究内容
围绕“望诊（舌象）+问诊（文本）”的跨模态证候识别与推荐，聚焦肺系疾病场景，构建“YOLO舌象检测—LLM前置语义解析—中置推理模型—LLM后置解释生成—三端一体化应用”的完整体系，以实现从数据到应用的闭环验证。该总体思路与前期开题与选题文件中提出的技术路线与数据来源一致，舌象数据统一至标准标签空间，系统形态为医生端、患者端、管理员端“三端一体”架构。
1.5.1 基于YOLO的舌象检测与特征提取
基于改进YOLO模型实现舌体区域分割、舌苔厚薄与颜色、舌质色与纹理、裂纹与齿痕等关键目标的检测与量化，输出标准化舌象特征向量，为后续推理提供客观影像依据。
1.5.2 前置LLM语义解析与中置推理建模
以中文通用大语言模型作为前置解析器，将患者主诉与问诊文本标准化为结构化症状特征；融合舌象向量，训练名医辅助辨证模型对肺病证候进行多标签判别并对候选方剂进行排序与推荐。
1.5.3 可解释性增强与三端一体化应用系统

1.6.4 前置大语言模型的问诊文本语义解析方法
针对患者主诉与问诊自由文本“非结构化、术语多变、存在歧义”的特点，本文引入中文通用大语言模型作为前置解析器，将文本内容约束输出为JSON Schema形式的结构化症状表示（包含症状名、起病时间、诱因、伴随症状、否定症状等字段），并引入同义/缩写表与冲突解析规则以提升一致性。
为保证系统稳定性，当模型置信度较低或命中黑名单词表时，系统回退到“词典+规则”的抽取器，实现可控的鲁棒性增强策略，从而降低纯生成式解析可能带来的不可控输出风险。
1.6.5 多模态融合与中置辨证推理建模方法
在得到结构化症状特征与舌象特征向量后，本文将二者进行融合建模，并嵌入“名医辅助辨证模型”，以实现证候的智能判别与诊疗建议生成（包括候选方剂排序推荐等）。
工程实现上，通过FastAPI将中置推理模块封装为可复用的RESTful服务接口，前端侧将舌象特征向量与语义解析输出拼接为请求输入，获取证候预测结果与候选方案，形成端到端可调用的推理链路。
1.6.6 后置解释生成与可解释性增强方法
为增强模型输出的可解释性与临床可审阅性，本文采用“后置LLM+模板化生成”的方式，将推理结果转化为条目化、可核实的解释文本。解释模板按照“诊断结论→辨证要点→舌象/症状证据→图谱依据三元组→方剂组成与功效→用药注意/禁忌”的结构组织输出。
同时，在提示词约束上限定解释只能引用“模型数值、图谱三元组与原始要点”，禁止外推未验证结论，从而在生成式表达与临床可控性之间取得平衡。

在业务流程接口方面，围绕病例创建、流水线触发、结果获取、医生审核提交、健康检查等关键环节设计标准化API，并结合JWT权限控制、图片脱敏、访问日志留痕等机制提升系统可审计性与可维护性。

2 开发技术和开发环境
2.1 系统开发环境
本系统采用前后端分离的开发模式，在通用桌面操作系统环境下完成系统开发与调试。系统开发过程中主要使用Python与JavaScript/TypeScript语言，结合主流Web开发框架与关系型数据库管理系统，实现中医智能辅助诊疗系统在业务逻辑处理、数据管理及智能分析等方面的功能开发。
系统开发环境主要包括操作系统、集成开发工具、数据库系统及相应运行环境。其中，操作系统基于macOS平台，后端开发主要使用PyCharm作为集成开发环境，以提升代码编写与调试效率；数据库采用MySQL 9.4.0进行数据存储与管理。整体开发环境配置稳定，具备良好的跨平台适应能力，能够满足系统开发、测试及运行的基本需求。
2.2 后端开发技术
系统后端采用基于Python的Web技术体系进行开发，核心框架选用Django，并结合Django REST Framework构建面向前端的接口服务。
Django框架具有结构清晰、组件完善和安全性较高等特点，适合实现用户管理、病例管理、诊疗流程控制等核心业务逻辑。通过Django的应用模块机制，可以将系统功能按业务领域进行拆分，提升系统的可维护性和扩展性。
在接口层面，引入Django REST Framework以RESTful风格对外提供数据服务，实现前后端分离条件下的统一数据交互。该框架支持序列化、权限控制及请求响应规范化处理，能够满足系统在多端访问和多角色协同场景下的接口需求。
此外，针对医患之间的即时消息推送与端侧红点未读提示场景，系统后端未采用厚重的WebSocket架构，而是引入了Server-Sent Events (SSE)单向长连接技术。借助Django的StreamingHttpResponse配合单机内存缓存构建微型事件总线，以极小的服务器性能损耗实现了高并发下的实时数据流推送，有效保障了医疗多端协同时的通讯时效性。
2.3 身份认证与安全机制
为保证系统的安全性与访问控制需求，本系统采用基于JSON Web Token（JWT）的身份认证机制。用户在登录后获取身份令牌，后续请求通过携带令牌完成身份校验，从而实现无状态的访问控制。
通过对不同用户角色进行权限划分，系统能够有效限制患者、医生与管理员对功能模块和数据资源的访问范围，保障系统数据安全和业务隔离性。该认证方式适用于前后端分离架构，具有良好的扩展性和跨端支持能力。
2.4 数据库与数据管理技术
系统采用关系型数据库进行数据存储与管理，用于保存用户信息、病例数据、舌象图像相关信息、问诊结构化结果、诊疗分析结果以及系统运行日志等内容。
关系型数据库在数据一致性、事务处理及复杂关联关系表达方面具有明显优势，能够满足中医智能辅助诊疗系统在多角色协同和流程流转过程中的数据管理需求，为系统的稳定运行提供可靠的数据支撑。
2.5 智能分析相关技术
在智能分析方面，系统引入计算机视觉与自然语言处理相关技术，对中医舌诊图像和问诊文本信息进行辅助分析。
舌象分析部分基于目标检测与图像处理技术，实现舌体区域定位及舌象特征提取，为中医舌诊的客观化分析提供基础支持。问诊文本处理部分结合自然语言处理与大语言模型技术，对患者主诉及问诊信息进行语义解析，生成结构化症状特征表示，并为后续证候推理与诊疗建议生成提供输入。
2.6 前端开发技术
2.6.1 医生端与管理员端前端技术
医生端与管理员端前端技术 医生端与管理员端前端采用基于React体系的全栈框架Next.js（基于App Router架构）进行开发。通过React Hooks及组件化思想实现复杂业务界面的组织与复用；在UI视觉构建上，引入Tailwind CSS原子化样式引擎搭配Shadcn UI高级业务组件库，实现响应式与深色模式的工作台快速搭建；在数据统计可视化方面，集成 Recharts 图表库实现多维度数据的清晰渲染。前端框架内置的网络优化与路由预加载能力，极大满足了医生进行高频次病例审核与管理员监测系统看板的性能需求。
2.6.2 患者端前端开发技术
患者端前端采用微信小程序原生框架进行开发，使用TypeScript作为主要编程语言，在提升开发效率的同时有效保障了代码的类型安全与可维护性。界面视图层基于小程序原生的WXML与WXSS标准进行构建，定制化实现了应用首页、病例创建、辅助诊断详情以及个人中心等主要功能页面。借助微信小程序轻量化、即开即用的生态特性，患者端无需额外安装即可顺畅完成舌象图像采集上传、问诊信息录入及分析结果查询等核心交互，极大地降低了用户设备的使用门槛与操作成本。

本系统面向“舌诊客观化+问诊语义结构化+多模态融合辨证+结果可解释”的中医智能辅助诊疗场景，需求分析重点围绕三端协同（患者、医生、管理员）、诊疗流水线闭环（采集—分析—展示—审核—留痕）以及安全合规与可追溯性展开。系统既要缓解传统舌诊主观性强、缺乏统一量化标准等问题，又要弥补对问诊文本语义信息利用不足、融合与可解释性薄弱等痛点。

构建一套融合舌象视觉信息与问诊文本语义信息的中医智能辅助诊疗系统，实现从病例采集、智能分析、结果展示到医生审核与系统管理的闭环流程，提升舌诊的客观化水平与辨证辅助效率，并通过条目化、可追溯的依据说明增强系统输出的可解释性与可信度。

管理员侧：对系统运行与数据治理项进行管理维护，并具备健康检查与审计追踪能力。

系统需满足面向实际使用的基本性能要求：
支持并发访问下的稳定响应，保证病例查询、列表加载等常规操作的交互流畅；
对智能分析流程采用任务化/流水线式组织，支持进度查询与结果异步获取，避免长耗时请求造成阻塞；
具备一定的可扩展性，便于后续替换/升级舌象检测、语义解析与推理服务。
（4）安全目标
系统需保证数据访问安全与业务隔离：
采用统一身份认证与权限控制机制，区分患者、医生、管理员的访问范围；
对关键业务操作保留访问与审计记录，支持事后追溯；
在接口层面保证数据传输与访问控制的规范性，降低越权访问风险。
3.2.2 信息系统需求结构
（1）需求分析过程与结论
系统需求来源主要包括三类：
1）文献与网络调研：结合“中医舌诊智能化、YOLO在医学视觉任务中的应用、大语言模型语义解析与可解释生成”等相关研究进展，明确本系统应支持舌象分析与问诊语义解析的协同。
2）问题导向分析：围绕舌诊主观性强、复杂环境适应不足、问诊语义利用不足、多模态融合与可解释性欠缺等问题，提炼出“客观化、融合化、可解释、可追溯”的核心需求。
3）业务流程抽象：以病例为中心，将“采集—分析—展示—审核—管理”抽象为系统闭环流程，并落实到病例创建、流水线运行、结果展示、医生审核与管理维护等关键环节。
综合分析结论为：系统需要以“病例”为主线组织数据与流程，以“流水线任务”承载智能分析步骤，以“三端协同”承载临床使用与管理治理，并通过解释与审计机制提升可信度与可追溯性。

3.4 性能分析
为了保证本系统在实际多端（患者端、医生端、管理员端）交互场景下能够稳定、流畅地运行，特别是针对涉及图像视觉分析和大语言模型语义推理的耗时任务，对系统的性能需求提出如下分析与要求：
3.4.1 系统响应时间要求
常规业务交互：系统的前端页面（包括微信小程序与React管理后台）的静态资源加载时间应控制在2秒以内。对于常规的CRUD操作（如病例增删改查、用户资料更新、数据字典加载等），接口响应时间应小于1秒，从而保证医生、患者和管理员在日常操作中的交互流畅性。
异步计算与智能分析：由于舌象图像的YOLO视觉分析与大语言模型推理均属于高算力、长耗时操作，系统不对该类请求的同步完成时间设限，但要求在用户发起诊断请求后，系统必须在立即返回“任务已接纳”状态，并通过流水线进度查询接口实时反馈当前任务的处理阶段（例如：图像提取中、特征分析中、报告生成中）。
3.4.2 并发处理与吞吐量需求
多并发支持：考虑到系统可能被多名患者同时上传病例及多位医生同时在线审阅，后端系统需具备一定的并发处理能力。Web服务器及数据库应能够支持百级别以上的用户并发访问而不出现请求丢失或宕机现象。
任务队列缓冲：对于瞬时高并发的智能分析请求，系统需具备缓冲机制（如采用异步任务队列），保证后台模型推理服务不会因超载而崩溃，任务能够按照优先级和提交顺序被稳定调度和处理。
3.4.3 系统稳定性与资源利用率
长期稳定运行：系统需满足7×24小时连续运行而不发生内存泄漏或进程意外退出的要求。特别是针对后台的AI推理服务，需具备服务状态监控及进程自动重启的容错能力。
可扩展性：系统的架构设计需保证计算节点的可扩展性。随着用户量及病例数据的累积，支持后续通过增加服务器节点、配置负载均衡，或是为YOLO模型与大语言模型接入独立的GPU算力节点，以实现推理能力的平滑扩容。
3.5 安全分析
本系统的核心数据涵盖患者病史、舌象图像等隐私信息，以及模型推理产生的辅助诊断建议，因此在系统架构设计层面需充分考虑运行安全与数据保护。
首在数据存取与访问层面，系统对三端用户（患者、医生、管理员）实行严格的基于角色的访问控制（RBAC）机制与接口Token鉴权。患者仅能访问个人病历数据，医生由于审核需要可操作分配的全部病例，而管理员仅拥有系统配置与运行维护权限，禁止越权操作涉及诊疗核心的业务数据。用户上传的敏感生物特征（舌体图像），其访问均支持匿名化及脱敏处理。
其次在系统运行与追溯层面，为防止接口被恶意攻击或大规模自动化调用消耗AI算力，核心API进行了流量速率限制与频率拦截。同时，为了保证辅助诊疗流程的可信度与后续可审阅性，系统对诊疗结论的修改、驳回及管理员的配置变动均引入了核心业务日志，并进行不可逆的审计留痕，为潜在的医疗责任判定提供准确的操作记录依据。前后端的数据交互全过程亦通过HTTPS协议保证传输安全。 
4 系统设计
4.1 系统设计原则
4.1.1 分层架构原则
本系统采用前后端分离的分层架构设计，将系统划分为表示层、业务逻辑层和数据访问层，各层职责明确、相互独立。表示层由医生端Web前端（Next.js）和患者端微信小程序构成，负责用户交互与界面呈现；业务逻辑层由Django REST Framework构建的后端服务负责，封装核心业务规则、权限控制与AI流水线调度；数据访问层以MySQL数据库持久化所有结构化数据。三层之间通过RESTful JSON API进行通信，降低层间耦合度，便于独立开发与维护。
4.1.2 模块化设计原则
系统后端按业务领域划分为accounts（用户与认证）、cases（病例管理）、pipeline（AI流水线）、chat（医患即时通讯）、notifications（系统通知）、followups（用药随访）、audit（操作审计）等独立Django App，每个模块对外暴露统一的REST API接口。模块间通过接口依赖而非直接调用内部实现，保证了高内聚、低耦合。
4.1.3 安全性原则
系统采用JWT（JSON Web Token）机制实现无状态身份认证，所有API请求均须携带有效令牌。权限控制基于角色（RBAC）实现，分为患者、医生、管理员三种角色，每个接口根据操作敏感程度配置相应的角色权限校验。密码使用Django内置的PBKDF2哈希算法存储，敏感操作（登录、病例创建、医生审核）同步写入审计日志，确保操作可追溯。
4.1.4 可靠性原则
AI流水线（舌象检测、证候辨别、处方推荐、LLM解释）采用Celery异步任务队列执行，避免因AI推理耗时导致HTTP请求超时；前端通过轮询方式查询流水线进度，实时向用户反馈处理状态。医患实时聊天采用Server-Sent Events（SSE）长连接推送，确保消息的低延迟送达。网络请求层实现了自动重试机制，提高系统在弱网环境下的可用性。
4.1.5 可扩展性原则
数据库核心实体（病例、用户）均采用UUID主键，避免整型自增ID在分布式或迁移场景下的冲突问题。AI流水线各阶段以函数形式独立封装，可根据需要增减或替换推理节点。前端组件采用shadcn/ui+Tailwind CSS设计系统，支持主题定制和组件复用。系统预留了admin角色及专属管理后台，便于后续扩展运营管理功能。
4.1.6 用户体验原则
界面设计遵循一致性原则，医生端Web使用深色模式兼容的现代UI风格，患者端小程序采用符合微信生态设计规范的绿色主题。关键操作（如AI诊断提交、病例审核）设计二次确认环节，防止误操作。错误提示均展示来自后端的具体业务原因，而非原始HTTP状态码，提升用户的操作引导体验。

5 UI设计
本系统面向不同使用场景和用户群体（普通患者与专业医师），在UI（用户界面）设计上采取了“端侧定制、体验隔离”的差异化策略。患者端运行于微信小程序，遵循微信体系轻量、直观的规范，采用大色块与引导式交互；医生及管理员端则为桌面大屏显示设计，侧重于高密度信息的并排呈现与深色严肃的工作台模式。本章详细阐述核心界面的设计思路与交互流程。
5.1 患者端核心UI设计
患者端主要服务于普通大众，界面设计强调“操作零门槛”与“流程高透明”，以绿色（代表健康、中医）与白色为主色调，确保视觉清晰、舒适。
5.1.1 首页与状态汇总台设计
设计思路：首页是患者打开小程序的“第一落脚点”，承担着核心服务入口与当前状态汇总的双重职责。顶部区域采用大面积卡片设计，放置最醒目的“开始智能诊断（新建病例）”按钮，引导用户发起核心业务。中间部分以横向滑动数据看板的形式，直观展示用户当前“待处理”、“诊断中”、“已完成”的病例数字概览，降低用户的查询焦虑。下方以列表形式按时间倒序平铺最近的病例卡片，每张卡片展示病历快照（主述摘要、主诊医生、当前状态标签），状态标签通过红、黄、绿等颜色区分，使用户对病程进展一目了然。
 
图5-1 患者端小程序-首页概览图
5.1.2 结构化问诊与舌象采集界面设计
设计思路：新建病例是系统最核心的输入环节。为避免长表单带来的页面折叠与输入疲劳，在此处采用了“渐进式分步卡片”的设计。第一部分采用下拉列表与标签点击的方式让用户选择科室与主诊医生；第二部分用富文本框收集自由主诉；第三部分将中医问诊简化为结构化的必填项（如二便、睡眠等）。最关键的舌象采集区位于页面底部，采用大尺寸占位符并在中央设置巨大的“+”按钮，支持调用系统相机或相册。为了提示用户上传高质量照片，在此版块周围添加了文字气泡形式的“光线与对焦说明”，从而从源头上提升后续YOLO模型的检测成功率。
   
图5-2 患者端小程序-新建病例/舌象采集页面图
5.1.3 AI报告详情与交互动作区设计
设计思路：当病例处于AI诊断完成尚未正式提交给医生的阶段时，详情页需承担起“让患者审阅并确认”的作用。页面采用“总-分-总”结构：顶部为患者的基本信息快照与固定于顶部的当前状态流水条；中段大量运用白底圆角的卡片，首个卡片展示原始上传图与AI给出的检测切割图（视觉对比直观呈现AI介入感），后续卡片利用高亮加粗字体逐条列出AI提取的“舌色、舌苔特征”以及由大语言模型总结的“辨证参考”；底部常驻悬浮栏（吸底组件），提供一个醒目且独立的主按钮——“确认提交给医生审核”，并在按钮上方添加免责与提示文本，确保患者产生明确的“阅后提交”动作。

5.2 医生端核心UI设计
医生端部署于PC桌面环境，面向医疗专业人员的日常高频操作。整体设计采用类似现代电子病历（EMR）的左侧导航、右侧多视图并排布局，色调基于“暗黑/冷灰深色模式”搭建，减少长时间盯屏疲劳，并使病理图片（通常色彩鲜艳）在深色背景下获得更高的对比度。
5.2.1 医生病例管理工作台
设计思路：医生登录后的主控台（Dashboard）设计追求“信息密度最大化”。页面顶端横向排布四个关键指标（KPI）状态卡片图表（如待审总数、今日已审、高频证候排行），让医生瞬时掌握工作负荷。页面主体被一个庞大的Data Table（数据表格）占据，用来呈现所有流转至该医生的病例队列。设计上弱化了表格的竖向边框，通过斑马纹与悬停高亮增加横向行的阅读连贯性。最重要的是，表格上方配置了“草稿、诊断中、待审核、已通过、已驳回”的Tab组分类器（Tabs Component），医生只需一键点击即可切换队列。其中，“待审核”Tab会被附加上红色数字角标，制造视觉焦点，催促医生尽快处理积压任务。

7.1 用户注册与登录认证模块

# 截取自：backend/apps/accounts/views.py
from rest_framework import status
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework.permissions import AllowAny
from rest_framework_simplejwt.tokens import RefreshToken
from .serializers import LoginSerializer, UserSerializer
from apps.audit.utils import log_action
class LoginView(APIView):
    """
    用户登录接口 (POST /auth/login/)
    """
    permission_classes = [AllowAny] # 登录接口对外完全开放
    def post(self, request):
        serializer = LoginSerializer(data=request.data)
        serializer.is_valid(raise_exception=True)
        
        # 提取校验通过的安全 User 实体
        user = serializer.validated_data['user']
        
        # 使用 SimpleJWT 库为当前用户对象签发非对称加密的 JWT Token 对
        refresh = RefreshToken.for_user(user)
        access_token = str(refresh.access_token)
        
        # 旁路记录安全审计日志，留存登录 IP 等网络指纹
        log_action(user=user, action='login', details={'ip': self.get_client_ip(request)})
        return Response({
            'token': access_token,
            'refresh': str(refresh),
            'role': user.role,
            'user': UserSerializer(user).data
        }, status=status.HTTP_200_OK)
    def get_client_ip(self, request):
        """防逆向获取经过 Nginx 等代理层转派的真实客户端 IP"""
        x_forwarded_for = request.META.get('HTTP_X_FORWARDED_FOR')
        if x_forwarded_for:
            return x_forwarded_for.split(',')[0]
        return request.META.get('REMOTE_ADDR', '')
7.2 核心AI智能辅助诊断调度模块
本系统的核心难点在于跨越多个神经网络架构（YOLO视觉模型、机器学习决策树与通用大语言模型）进行聚合推理，这是极度耗时的重负载作业。若采用传统的同步HTTP请求响应模型，必将导致Web网关框架超时。因此系统对于AI模型的推理任务进行了独立线程下放，并将其精细切分为四个必须顺次执行的Pipeline（流水线）子任务阶段，期间不断打点更新progress字段供前端轮询动画。
第一阶段（CV Phase）：下发YOLO引擎提取舌体特征图像，耗时进度10%到40%
第二阶段（NLP Phase）：下发大模型进行非结构化长文本（主诉病情）分析提取出结构化的中医症状集，耗时进度40%到60%
第三阶段（Inference Phase）：使用中医机器学习特征分类器，统合视觉与文本节点，预测患者证候与推荐用药，耗时进度60%到85%
第四阶段（Explanation Phase）：再次调用大模型将所有中医术语转化为白话文解释，最终组装返回，耗时进度85%到100%

        pipeline_run.yolo_result_json = yolo_result

        pipeline_run.save()
        # ==== 阶段 2: LLM 症状提取 (40% -> 60%) ====
        stage_start = time.time()
        pipeline_run.progress = 50
        pipeline_run.save()
        nlp_result = nlp_client.parse(case.chief_complaint_text, case.questionnaire_json)
        pipeline_run.nlp_result_json = nlp_result
        pipeline_run.progress = 60
        pipeline_run.save()
        # ==== 阶段 3: 中医 ML 决策辨证推理 (60% -> 85%) ====
        stage_start = time.time()
        pipeline_run.progress = 70
        pipeline_run.save()
        symptoms = nlp_result.get('symptoms', [])
        tongue_features = yolo_result.get('tongue_features', {})
        yolo_label = yolo_result.get('detections', [{}])[0].get('class_name', '')
        # 融合前置节点数据推入机器学习推断网关
        inference_result = syndrome_client.infer(symptoms, tongue_features, yolo_label)
        pipeline_run.inference_result_json = inference_result
        pipeline_run.progress = 85
        pipeline_run.save()
        # ==== 阶段 4: LLM 白话文综合分析解释 (85% -> 100%) ====
        stage_start = time.time()
        pipeline_run.progress = 90
        pipeline_run.save()
        syndromes = inference_result.get('syndromes', [])
        herbs     = inference_result.get('herbs', [])
        # 利用大模型进行知识图谱扩展降维解释
        explanation_text = explanation_client.generate(
            syndromes=syndromes, herbs=herbs, symptoms=symptoms,
            chief_complaint=case.chief_complaint_text, yolo_label=yolo_label,
        )
        pipeline_run.explanation_text = explanation_text
        pipeline_run.progress = 100
        pipeline_run.status = 'success'
        pipeline_run.save()
    except Exception as e:
        pipeline_run.status = 'failed'
        pipeline_run.error_message = str(e)
        pipeline_run.save()
7.3 业务边界网关与审核状态机引擎
在AI引擎处理完毕并交还报告后，患者核对无误将触发前端的独立接线口转交实际医生进行确认拦截。后端使用状态机约束以实现严格的网关控制。
以下为病例审核扭转路由控制阀的代码截取：
# 截取自：backend/apps/cases/views.py
from rest_framework import viewsets, status
from rest_framework.response import Response
from rest_framework.decorators import action
from .models import Case
from apps.audit.utils import log_action
from apps.notifications.utils import notify_doctors_new_case
class CaseViewSet(viewsets.GenericViewSet):
    """病例核心业务调度视图集"""
    @action(detail=True, methods=['post'], url_path='submit-to-doctor')
    def submit_to_doctor(self, request, pk=None):
        """
        AI 管线解析完毕后，患者将报告转交真实医生审核
        状态机流转：running (AI已完工) -> pending_review (医生待审)
        """
        try:
            case = Case.objects.get(pk=pk)
        except Case.DoesNotExist:
            return Response({'detail': '病例实体丢失'}, status=status.HTTP_404_NOT_FOUND)
        # 严格的业务约束边界：防御越权和状态机错乱
        if request.user.role == 'patient' and case.patient != request.user:
            return Response({'detail': '越权越界访问'}, status=status.HTTP_403_FORBIDDEN)
        if case.status != 'running':
            return Response({'detail': '不满足提交流水线的前置状态节点'}, status=status.HTTP_400_BAD_REQUEST)
        # 挂载状态扭转操作并唤醒通知模块
        case.status = 'pending_review'
        case.save()
        notify_doctors_new_case(case)
        log_action(user=request.user, action='case_submit', case=case)
        return Response({'ok': True, 'status': 'pending_review'})
7.4 实时医患沟通与消息推送模块
在医患即时沟通环节，若采用高频定时长轮询势必给Django集群施加大量空转查询压力；而部署完整的WebSockets/ASGI又太过笨重且受限于某些小程序的原生网络隔离池。为取得最佳平衡，后端最终抛弃了传统心跳模式，针对性采用Server-Sent Events（SSE）长链接搭配基于内存缓存的单机事件总线（Event Bus）分发机制。
其原理利用了Django的StreamingHttpResponse。该响应对象在生命周期内使用循环挂起（Generator Yield），只有当底库中检测到属于当前聊天室的全局Timestamp突发变动时，才组装新的增量JSON包推进网络流的客户端挂机口中。
以下为推送医患实时聊天消息的独立路由房间组件实现：
# 截取自：backend/apps/chat/views.py
import json
import time
from rest_framework.views import APIView
from django.http import StreamingHttpResponse
from django.core.cache import cache
from .models import ChatMessage
from .serializers import ChatMessageSerializer
class ChatStreamView(APIView):
    """
    负责保持单一会话的 Server-Sent Events 长链接房间通道
    一旦监听到新的 ORM Hook，将立即释放挂机并下发增量包裹
    """
    def get(self, request, patient_id):
        def event_stream():
            # 划定当前频道的广播订阅 ID 并对齐世界观时间戳
            cache_key = f'chat_update_{patient_id}'
            yield "data: {\"type\": \"connected\"}\n\n"
            last_timestamp = cache.get(cache_key) or time.time()
            if cache.get(cache_key) is None:
                cache.set(cache_key, last_timestamp, timeout=None)
            last_message_id = ChatMessage.objects.filter(patient_id=patient_id).last().id or 0
            # 无限挂起轮询守护池
            while True:
                time.sleep(1) # 限流控制，避免 CPU 100% 飙升
                current_timestamp = cache.get(cache_key)
                # 若检测出世界线变动，触发捞取游标后面的增量载荷
                if current_timestamp is None or current_timestamp != last_timestamp:
                    new_messages = ChatMessage.objects.filter(
                        patient_id=patient_id, 
                        id__gt=last_message_id
                    ).order_by('created_at')
                    if new_messages.exists():
                        data = json.dumps({'type': 'new_messages', 'messages': ChatMessageSerializer(new_messages, many=True).data})
                        yield f"data: {data}\n\n"  # 规范发送 SSE 报文
                        last_message_id = new_messages.last().id 
                        
                    last_timestamp = current_timestamp
                else:
                    # 保活心跳 ping，防止被 Nginx 掐平
                    yield ": ping\n\n"
        # 发送特殊的关闭 Buffer 的 Header 保证推流可以实时刷出
        response = StreamingHttpResponse(event_stream(), content_type='text/event-stream')
        response['Cache-Control'] = 'no-cache'
        response['X-Accel-Buffering'] = 'no'
        return response
7.5 管理端超参配置与健康巡检群
作为系统底座，监控集群对内网数据库及大模型外网API是否出现波动也是工程中不可或缺的一环。系统在Django自身中内嵌了一套极轻量的探针阵列（Probe Array）。这个微服务健康检索服务可以在一次HTTP生命周期内，并发侦测数据库MySQL连通性、YOLO权重文件（.pt）、ML推理模型权重文件（.pkl）完整性及内存载入验证，并通过预发Request向云端硅基流动平台的私密模型网关校验当前的全局Secret环境变量。
以下展示了后端聚合检查引擎的设计逻辑层：
# 截取自：backend/apps/adminops/views.py
import os
import time
import requests
from datetime import datetime
from rest_framework.views import APIView
from rest_framework.response import Response
class HealthCheckView(APIView):
    """
    服务健康检查聚合测试接口 (GET /api/admin/health/)
    """
    def get(self, request):
        """汇总各子群服务的并集体量包，抛出网络木桶短板"""
        services = [
            self._check_django(),
            self._check_database(),
            self._check_yolo_model(),  # 检查服务器本地 CV 依赖权重是否丢失
            self._check_ml_models(),   # 检查证型/中药模型 pkl 是否存在
            self._check_llm_api(),     # 自测发包校验 LLM 云端联通性
        ]
        overall_status = self._calculate_overall_status(services)
        return Response({'services': services, 'overall_status': overall_status})
    
    def _check_llm_api(self) -> dict:
        """探针：检查外部 LLM API 外网节点连通度及私钥有效性"""
        start_time = time.time()
        try:
            # 动态抽离当前保存在数据库的环境变量池（实现无重启热插拔重配模型）
            from apps.adminops.models import AIConfig
            api_url = AIConfig.get('llm_api_url')
            api_key = AIConfig.get('llm_api_key')
            model   = AIConfig.get('llm_model_name')
        except Exception:
            api_url, api_key, model = '', '', ''
        try:
            # 向目标外部平台发起极低能耗下的对话连线侦测
            resp = requests.post(
                api_url,
                headers={'Authorization': f'Bearer {api_key}', 'Content-Type': 'application/json'},
                json={
                    'model': model,
                    'messages': [{'role': 'user', 'content': 'ping'}],
                    'max_tokens': 5, # 将计费Token损耗截断在5字以内
                },
                timeout=10 # 最高挂起 10s 预先爆破避免级联雪崩
            )
            latency_ms = int((time.time() - start_time) * 1000)
            if resp.status_code == 200:
                return {'name': 'LLM API', 'status': 'healthy', 'latency_ms': latency_ms, 'message': '接口稳定'}
            else:
                return {'name': 'LLM API', 'status': 'unhealthy', 'latency_ms': latency_ms, 'message': resp.json().get('message')}
        except Exception as e:
            return {'name': 'LLM API', 'status': 'unhealthy', 'message': str(e)[:80]}
