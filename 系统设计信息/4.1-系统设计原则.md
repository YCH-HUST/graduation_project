# 4.1 系统设计原则

## 1. 分层架构原则

本系统采用前后端分离的分层架构设计，将系统划分为表示层、业务逻辑层和数据访问层，各层职责明确、相互独立。表示层由医生端 Web 前端（Next.js）和患者端微信小程序构成，负责用户交互与界面呈现；业务逻辑层由 Django REST Framework 构建的后端服务负责，封装核心业务规则、权限控制与 AI 流水线调度；数据访问层以 MySQL 数据库持久化所有结构化数据。三层之间通过 RESTful JSON API 进行通信，降低层间耦合度，便于独立开发与维护。

## 2. 模块化设计原则

系统后端按业务领域划分为 accounts（用户与认证）、cases（病例管理）、pipeline（AI流水线）、chat（医患即时通讯）、notifications（系统通知）、followups（用药随访）、audit（操作审计）等独立 Django App，每个模块对外暴露统一的 REST API 接口。模块间通过接口依赖而非直接调用内部实现，保证了高内聚、低耦合。

## 3. 安全性原则

系统采用 JWT（JSON Web Token）机制实现无状态身份认证，所有 API 请求均须携带有效令牌。权限控制基于角色（RBAC）实现，分为患者、医生、管理员三种角色，每个接口根据操作敏感程度配置相应的角色权限校验。密码使用 Django 内置的 PBKDF2 哈希算法存储，敏感操作（登录、病例创建、医生审核）同步写入审计日志，确保操作可追溯。

## 4. 可靠性原则

AI 流水线（舌象检测、证候辨别、处方推荐、LLM解释）采用 Celery 异步任务队列执行，避免因 AI 推理耗时导致 HTTP 请求超时；前端通过轮询方式查询流水线进度，实时向用户反馈处理状态。医患实时聊天采用 Server-Sent Events（SSE）长连接推送，确保消息的低延迟送达。网络请求层实现了自动重试机制，提高系统在弱网环境下的可用性。

## 5. 可扩展性原则

数据库核心实体（病例、用户）均采用 UUID 主键，避免整型自增 ID 在分布式或迁移场景下的冲突问题。AI 流水线各阶段以函数形式独立封装，可根据需要增减或替换推理节点。前端组件采用 shadcn/ui + Tailwind CSS 设计系统，支持主题定制和组件复用。系统预留了 `admin` 角色及专属管理后台，便于后续扩展运营管理功能。

## 6. 用户体验原则

界面设计遵循一致性原则，医生端 Web 使用深色模式兼容的现代 UI 风格，患者端小程序采用符合微信生态设计规范的绿色主题。关键操作（如 AI 诊断提交、病例审核）设计二次确认环节，防止误操作。错误提示均展示来自后端的具体业务原因，而非原始 HTTP 状态码，提升用户的操作引导体验。
