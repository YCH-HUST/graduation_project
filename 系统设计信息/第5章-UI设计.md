# 第5章 UI设计

本系统面向不同使用场景和用户群体（普通患者与专业医师），在UI（用户界面）设计上采取了“端侧定制、体验隔离”的差异化策略。患者端运行于微信小程序，遵循微信体系轻量、直观的规范，采用大色块与引导式交互；医生及管理员端则为桌面大屏显示设计，侧重于高密度信息的并排呈现与深色严肃的工作台模式。本章详细阐述核心界面的设计思路与交互流程。

## 5.1 患者端 UI 设计

患者端主要服务于普通大众，界面设计强调“操作零门槛”与“流程高透明”，以绿色（代表健康、中医）与白色为主色调，确保视觉清晰、舒适。

### 5.1.1 首页与状态汇总台设计
**设计思路**：
首页是患者打开小程序的“第一落脚点”，承担着核心服务入口与当前状态汇总的双重职责。顶部区域采用大面积卡片设计，放置最醒目的“开始智能诊断（新建病例）”按钮，引导用户发起核心业务。中间部分以横向滑动数据看板的形式，直观展示用户当前“待处理”、“诊断中”、“已完成”的病例数字概览，降低用户的查询焦虑。下方以列表形式按时间倒序平铺最近的病例卡片，每张卡片展示病历快照（主述摘要、主诊医生、当前状态标签），状态标签通过红、黄、绿等颜色区分，使用户对病程进展一目了然。

### 5.1.2 结构化问诊与舌象采集界面设计
**设计思路**：
新建病例是系统最核心的输入环节。为避免长表单带来的页面折叠与输入疲劳，在此处采用了“渐进式分步卡片”的设计。第一部分采用下拉列表与标签点击的方式让用户选择科室与主诊医生；第二部分用富文本框收集自由主诉；第三部分将中医问诊简化为结构化的必填项（如二便、睡眠等）。最关键的舌象采集区位于页面底部，采用大尺寸占位符并在中央设置巨大的“+”按钮，支持调用系统相机或相册。为了提示用户上传高质量照片，在此版块周围添加了文字气泡形式的“光线与对焦说明”，从而从源头上提升后续YOLO模型的检测成功率。
> **此处配图**：【患者端小程序 - 新建病例/舌象采集页截图】

### 5.1.3 AI报告详情与交互动作区设计
**设计思路**：
当病例处于AI诊断完成尚未正式提交给医生的阶段时，详情页需承担起“让患者审阅并确认”的作用。页面采用“总-分-总”结构：顶部为患者的基本信息快照与固定于顶部的当前状态流水条；中段大量运用白底圆角的卡片，首个卡片展示原始上传图与AI给出的检测切割图（视觉对比直观呈现AI介入感），后续卡片利用高亮加粗字体逐条列出AI提取的“舌色、舌苔特征”以及由大语言模型总结的“辨证参考”；底部常驻悬浮栏（吸底组件），提供一个醒目且独立的主按钮——“确认提交给医生审核”，并在按钮上方添加免责与提示文本，确保患者产生明确的“阅后提交”动作。

## 5.2 医生端工作台 UI 设计

医生端部署于PC桌面环境，面向医疗专业人员的日常高频操作。整体设计采用类似现代电子病历（EMR）的左侧导航、右侧多视图并排布局，色调基于“暗黑/冷灰深色模式”搭建，减少长时间盯屏疲劳，并使病理图片（通常色彩鲜艳）在深色背景下获得更高的对比度。

### 5.2.1 医生Dashboard与病例过滤工作台
**设计思路**：
医生登录后的主控台（Dashboard）设计追求“信息密度最大化”。页面顶端横向排布四个关键指标（KPI）状态卡片图表（如待审总数、今日已审、高频证候排行），让医生瞬时掌握工作负荷。页面主体被一个庞大的Data Table（数据表格）占据，用来呈现所有流转至该医生的病例队列。设计上弱化了表格的竖向边框，通过斑马纹与悬停高亮增加横向行的阅读连贯性。最重要的是，表格上方配置了“草稿、诊断中、待审核、已通过、已驳回”的Tab组分类器（Tabs Component），医生只需一键点击即可切换队列。其中，“待审核”Tab会被附加上红色数字角标，制造视觉焦点，催促医生尽快处理积压任务。
> **此处配图**：【医生端Web - Dashboard病例工作台与过滤Tab截图】

### 5.2.2 辅助诊断审阅与修订工作流设计
**设计思路**：
该页面是系统的核心功能落地场景（审阅页）。为充分利用PC端宽屏优势，采用了“左中右三栏分块”或“经典左右分栏”的对比式阅读布局。左栏固定为纵向滚动的报告源信息区，包含患者主诉、问卷细节以及高分辨率的舌象原图与模型圈注图，支持点击放大检视。右侧大面积区域留给核心的“审核面板”：包含模型推荐的处方、图谱三元组证据链及生成的文字解释。在此面板最下方，常驻三个功能明确的操作按钮组合——绿色的“审核通过（Approve）”、橙/黄色的“修订通过（Revise）”与红色的“驳回（Reject）”。若医生点击“修订”，面板相关文本框将从只读无缝切换为可编辑状态，支持医生动态删改证候名或中药药味，体现“AI打底、医生把关”的辅助诊疗内核。
> **此处配图**：【医生端Web - 病例详情审阅与通过/驳回/修改面板截图】

### 5.2.3 医患双向即时通讯窗口设计
**设计思路**：
为了方便医生在审阅病例时向患者补充询问，或解答日常用药疑惑，医生端配置了专属的聊天模块界面。设计上借鉴了主流生产力通讯软件（如Slack、企业微信）的界面范式：采用经典两栏分割结构。左侧较窄的长廊为“会话列表（Conversation List）”，以患者头像、姓名及最新一条消息摘要组成，未读消息以红点气泡标记并置顶排序。右侧为主干会话流（Chat Stream），采用典型的气泡对话框展现——深色/主题色气泡靠右表示医生发出的消息，浅灰色气泡靠左表示患者发来的消息，有效降低视觉混淆。底部为多行文本输入框与“Send”发送按钮，并且输入框支持响应式随打字行数自动增高。
> **此处配图**：【医生端Web - 医患聊天窗口（双分栏）截图】

### 5.2.4 管理员端运维与健康巡检仪表盘设计
**设计思路**：
系统的底层运行高度依赖数据库、Redis缓存及多项外部模型接口的稳定性，因此管理员端特别设计了集中的“健康检查（Health Check）”运维仪表盘。界面摈弃了传统后台的表单罗列，采用“雷达式扫描面板”风格。页面顶部中央放置一个大尺寸的主控按钮“执行系统全面体检（Run System Health Check）”。下方划分为多个网格化状态卡片（如：Database、Redis、YOLO Service、LLM API、Machine Learning Models 等）。当检查运行时，各卡片会展示加载动画；结果返回后，通过红（异常）、绿（正常）、黄（延迟高）三种醒目的状态灯颜色直观反馈该网元的通信健康度，同时在卡片背板直接透出测距的Ping值与简明错误日志。此设计使零代码基础的运营人员也能一秒定位系统瓶颈。
> **此处配图**：【管理员端Web - 系统健康检查扫描仪表盘截图】

### 5.2.5 AI模型与配置治理页面设计
**设计思路**：
为保证系统的算法推演具备灵活性，管理员端下设专属的“AI 配置”功能页。其设计思路遵循“高敏感度配置项折叠与防误触”原则。页面主要分为两块：一块是YOLO目标检测相关配置（如置信度阈值调优参数注入）；另一块是大语言模型（LLM）的接口参数配置（包含 Base URL替换、API Key 暗文输入框、System Prompt 固定话术修改池）。为防止核心 Prompt 被意外抹除，文本域的每次保存均加入了“双重验证弹窗（Double Confirm）”的交互卡点。整体采用卡片化的设置组排版，确保表单之间视觉互不干扰，在修改生效后会在页面顶部下发带有时间戳的Toast系统通报，以完成配置变更操作的交互闭环。
