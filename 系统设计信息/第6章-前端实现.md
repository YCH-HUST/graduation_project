# 第6章 前端实现

本章详细阐述中医智能辅助诊疗系统前端两端（患者端、医生端）的代码设计思路与技术落地细节。两端虽然基于不同的框架（微信小程序原生开发与 React/Next.js），但在组件拆分、状态管理以及与后端接口的对接上均遵循了高内聚低耦合的工程化标准。

## 6.1 患者端（微信小程序）实现

患者端注重轻量化与采集的便利性，采用 TypeScript 配合小程序原生架构进行开发。

### 6.1.1 舌象数据采集与预处理优化
患者端的最核心功能是采集并上传舌象图片。在代码实现上，通过调用 `wx.chooseMedia` API 唤起原生相机与相册。
为降低后端带宽压力与大文件传输导致的请求等待时间，我们在前端进行了第一道图片压缩（通过配置 `sizeType: ['compressed']`），并使用 `wx.getImageInfo` 获取图片长宽比进行边界合法性校验。获取合法临时路径后，我们并未直接将图片转为Base64阻断主线程，而是调用 `wx.uploadFile` 以 `multipart/form-data` 的流式格式分片上传至 Django 后端的媒体存储桶中，从而保证了上传过程的极速交互体验。

### 6.1.2 异步状态轮询与渲染逻辑
病例提交后，AI后台处理属于耗时操作（通常需要数十秒乃至数分钟）。为了防止界面卡死，前端在获取到 `case_id` 之后，立即跳转至病例详情预览页，并启动前端定时器（`setInterval`），每隔2秒向后端的病例查询接口发起长轮询（Long Polling）。
当接口返回的 `status` 字段从 `running` 变为 `pending_review` 或是直接附带了 `yolo_result_json` 节点包时，定时器自动销毁。此时页面基于数据驱动框架（如 `this.setData`）触发视图层（WXML）重绘，将原本占位的骨架屏（Skeleton Screen）替换为真实的舌象切割预览图与大语言模型的结构化结论。

## 6.2 医生端（PC Web）实现

医生端作为一个严肃的医疗工作台，对信息密度与实时性要求极高。采用 Next.js (App Router) 搭配 Shadcn/UI 组件库进行构建。

### 6.2.1 响应式布局与高阶组件封装
医生端Dashboard包含复杂的病历检索、状态切换展示。为了解决多重筛选状态（如科室、病例状态、患者姓名）组合带来的状态冗余，我们在顶层剥离出一个 Custom Hook `useCasesFilter` 统一接管所有检索字段的变化，并通过防抖（Debounce）技术触发底层的 Axios 实例对后端发起数据拉取。
针对复杂的UI结构，我们大量运用了组合式组件（Compound Components）的设计模式。例如将病例审核面板拆分为 `<CaseImagePreview />`（负责渲染高分辨率图与YOLO框标记）、 `<AIDiagnosisPanel />`（负责渲染大模型的条目化推理节点）与 `<ReviewActions />`（负责处理医生的修订与驳回提交逻辑）三个跨文件、低耦合的高阶组件，最后集中组装，使得整体代码的复用度显著提高。

### 6.2.2 Server-Sent Events (SSE) 实时交互实现
为了实现医患沟通时不刷页也能及时看到最新消息，以及系统红点未读数字的即时同步，前端接入了单向长连接通讯技术（Server-Sent Events）。
我们在全局 Layout 层实例化了原生的 `EventSource` 对象，订阅了后端 `/api/chat/stream/` 通道。利用 React内置的 Context API 与 `useEffect` 钩子函数，监听来自服务器的 `new_messages` 广播事件。一旦监听到新推送，React 立即驱动全局状态（如 Zustand store）进行更新，触发右下角悬浮条（Toast）弹出并在侧边栏聊天列表顶端动态追加新会话气泡。这一设计规避了 WebSocket 的重负载，又实现了秒级的双向沟通体验。
