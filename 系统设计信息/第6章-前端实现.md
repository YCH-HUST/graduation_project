# 第6章 前端实现

本章详细阐述了本系统中不同使用终端（患者端、医生端、管理员端）前端视觉效果与核心交互逻辑的落地实现。为了契合“端侧定制、体验隔离”的设计原则，患者端采用微信小程序原生语法（WXML+WXSS+TypeScript）开发，主打轻量敏捷与原生设备接口的深度集成；而医生端与管理员端则共用基于 Next.js（React 18）与 Tailwind CSS 搭建的高性能 Web 桌面端骨架，侧重于高密度的复杂表单处理、数据流防抖防漏以及多维度数据的全景可视化。

## 6.1 患者端前端实现（微信小程序）

患者端运行于微信环境，是整个系统“数据源头”的入口。在此处的开发着重解决了原生表单校验流转、高并发长连接状态处理、以及医患消息通信的字节流解析等技术难点。

### 6.1.2 异步追踪与诊断状态的双向轮询流
在患者点击提交后，后端的 YOLO 与 LLM 推理管道需要执行一定时间。为了防止界面假死并向患者反馈真实进度，前端建立了一套递归短轮询机制（Polling）。此机制利用 `setTimeout` 递归随时拉取后端 `progress` 字段驱动 UI 动画。

（此处插入：图 患者端-AI流水线动态进度加载界面图）

```typescript
// 截取自：miniprogram/pages/new-case/new-case.ts
    // 启动递归轮询监听后端任务管线
    startPolling(caseId: string) {
        let attempts = 0
        const maxAttempts = 60 // 定制安全边界，防止无限死循环

        const poll = async () => {
            if (attempts >= maxAttempts) {
                this.setData({ step: 'error', errorMessage: '处理超时' })
                return
            }
            try {
                // 探活后端流水线当前上下文状态
                const status = await getPipelineStatus(caseId)
                this.setData({ progress: status.progress, currentStage: status.current_stage || '' })

                if (status.result_available) {
                    this.setData({ step: 'success' })
                    // 诊断完成，平滑重定向至报告详情页
                    setTimeout(() => wx.redirectTo({ url: `/pages/case-detail/case-detail?id=${caseId}` }), 1200)
                } else if (status.status === 'failed') {
                    // ...异常与重试逻辑...
                } else {
                    attempts++; setTimeout(poll, 2000) // 未就绪，挂起2秒后重新投递探活
                }
            } catch {
                attempts++; setTimeout(poll, 2000) // 容错处理防单次抖动
            }
        }
        poll()
    },
```

### 6.2.2 全局红点气泡即时轮询与中心事件总线层
在 Next.js App Router 架构中跨组件更新状态较繁琐，且传统 SSE 在 Web 环境中极易受到 Nginx 反向代理层挂起后的假死影响。因此，前端在最顶层侧边栏（`Sidebar.tsx`）自研了一个高可靠短轮询进程（Polling）。它不仅管理红点数字，还通过原生 `window.dispatchEvent` 进行广播，使得如若用户在另一标签页点开了聊天室，其它地方的气泡能完成精准消杀（消红点）。

（此处插入：图 医生端-侧边栏与系统消息总线运转架构图）

```tsx
// 截取自：frontend/src/components/layout/Sidebar.tsx
export function Sidebar() {
    const [unreadCount, setUnreadCount] = useState(0)

    useEffect(() => {
        if (role !== 'doctor' || !token) return
        let isPolling = true

        const fetchAllUnread = async () => {
            if (!isPolling) return
            try {
                // 并发拉取各类状态机，解决瀑布流水线加载导致的白屏时长
                const [notifCount, chatCount] = await Promise.all([
                    getUnreadCount().catch(() => 0),
                    import('@/api/chat').then(m => m.getChatUnreadCount()).catch(() => 0)
                ])
                if (isPolling) {
                    setUnreadCount(notifCount)
                    setUnreadChatCount(chatCount)
                }
            } catch (err) { console.error(err) }
        }

        fetchAllUnread() // 即时唤起并埋下心跳包
        const intervalId = setInterval(fetchAllUnread, 5000)

        // 中间件级别的挂载：监听深水区组件发送来的强行对齐回调广播
        const handleLocalRead = () => { fetchAllUnread() }
        window.addEventListener('notifications-read', handleLocalRead)
        
        return () => { isPolling = false; clearInterval(intervalId); }
    }, [role, token])
}
```

## 6.3 管理员端前端实现（Web 桌面）

管理员端需要管理隐藏在服务黑盒下的系统接口矩阵，其核心设计逻辑围绕“即时运维验证”、“模型参数防护覆盖”以及“极具扩张性的海量数据网格处理能力”。

### 6.3.1 跨语言微服务矩阵并发探活中心
管理员系统健康巡检中心并非常规数据库呈现。前端通过触发一颗异步扫描探针（`getHealth`），指令触发后端同时向其附属的 YOLO 服务、硅基平台外网、消息缓存总线分发多路请求以统计延时。前端拿到综合矩阵数据后，再使用动态类名挂载（Dynamic Class Binding）对不同节点的延迟健康度赋予以绿、黄、红为主体的交互警示反馈。

（此处插入：图 管理员端-各类系统探活扫描雷达矩阵效果展示图）

// 截取自：frontend/src/app/admin/users/page.tsx
export default function AdminUsersPage() {

            setSelectedIds(new Set()) // 全额清空指针池

             // ...
        } finally {

        setSelectedIds(prev => {

            return next
