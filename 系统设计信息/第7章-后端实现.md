# 第7章 后端实现

本章重点阐述中医智能辅助诊疗系统后端的业务流引擎实现与核心功能落地。后端采用 Python 生态中成熟的 Django 框架，并结合 Django REST Framework 提供标准化的架构支撑，重点解决了大模型集成、异步任务调度及多端并发访问的稳定性问题。

## 7.1 系统鉴权与安全审计实现

医疗类系统对数据互访隔离与操作追溯有严苛的要求。
我们在 Django 默认的用户系统之上扩展了 `Role`（角色）字段，通过自定义 `IsPatient`、`IsDoctor` 等 DRF 权限类（Permissions），物理阻断了患者越权访问全量病例池的可能性。身份令牌采用无状态的 JWT（JSON Web Token）机制，并在 `settings.py` 中配置了 Token 过期时间与黑名单轮换策略。
为了应对医疗纠纷追溯需要，我们设计了一套轻量级的审计拦截器系统。当特定高危 API（如 `/api/review/approve/` 审核通过接口）被请求时，重写视图层的 `perform_update` 方法，将当时的 `User`、修改快照及核心参数同步定格写入 `AuditLog`（审计日志）数据表，保证了全流程行为的不可逆留痕。

## 7.2 AI流水分离并行与 Celery 异步调度

本系统的核心难点在于跨越多个神经网络架构（YOLO CV模型与通用大语言模型）进行聚合推理，这是极度耗时的重负载作业。若采用传统的同步 HTTP 请求响应模型，将不可避免地导致 Nginx 超时（504 Gateway Time-out）与网关线程池枯竭。
为此，我们在架构层面引入了 Redis + Celery 构成的分布式任务调度队列。当患者提交舌象后，Django 仅负责在数据库内生成一个初始状态为 `queued` 的 `PipelineRun` 记录并立刻向前端返回 200 OK，随即将真实的执行任务下放至 Celery Worker。
在 Worker 进程中，处理链路被精细地拆解为若干个解耦的 Task 子流程：
1. **舌象切割（CV Phase）**：调用封装好的 YOLO 推理引擎，提取图片中的舌体 Mask 坐标。
2. **文本解析（NLP Phase）**：由大模型对非结构化主诉文本提取结构化 JSON 症状集。
3. **辨证预测与生成（Inference Phase）**：融合 CV 与 NLP 输出节点，喂给最终的辅助诊断引擎生成药方与中医释义。
每完成一个 Task 子流程，Worker 都会主动更新数据库中的 `progress` 字段。这一基于消息驱动的削峰填谷设计，彻底释放了 Django 的主进程并发处理能力。

## 7.3 持久化连接池与 SSE 单向推送

医疗系统中常存在需要服务端主动提醒的业务（例如：医生处理完病例后，患者端需即时收到弹窗）。为了在低配置服务器上撑起尽可能多的长连接，我们舍弃了笨重的 WebSocket，转而采用了更轻量的 Server-Sent Events（SSE）。
我们利用 Django 内置的 `StreamingHttpResponse` 构建了阻塞式生成器机制，并将 Django 默认的单机进程缓存（LocMemCache）作为进程内的事件总线。每当底层 ORM 发生 `ChatMessage` 或 `Review` 表的 `post_save` 数据库信号时，会修改对应记录时间戳，SSE 循环检测到时间戳突变，即唤醒当前挂起线程并向所属客户端输出流写入形如 `data: {"type": "new_messages"}\n\n` 的规范数据包，随后挂起。
为避免长期保持连通状态带来的 MySQL 假死错误（`MySQLdb.OperationalError: 2006, MySQL server has gone away`），我们在 `settings.py` 的数据库配置项中显式设定了合理的 `CONN_MAX_AGE` 参数值，使框架有能力复用持久化连接槽并优雅地抛弃并重连“脏”会话，保障了后端服务的健壮性。
