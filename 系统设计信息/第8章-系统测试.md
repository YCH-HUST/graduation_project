# 第8章 系统测试

系统测试是保障代码从研发走向真实可用落地的关键阶段。本章阐述了我们针对中医智能辅助诊疗系统的三大核心参与方（患者、医生、系统管理员），进行的端到端（E2E）功能走查、后台AI流水线的边界异常测试以及弱网并发场景下的稳定性考核。通过科学严谨的测试矩阵拦截了绝大部分崩溃与越权隐患，保障了本项目达到预期可演示标准。

## 8.1 核心功能组件跑通测试

### 8.1.1 用户授权隔离与状态转换边界测试
- **患者端用例**：
  - 测试新用户通过手机号/微信身份令牌静默注册登录的成功率。
  - **异常模拟**：患者A尝试越权通过修改URL参数 `case_id` 去拉取属于患者B的病例（测试防御预期：后端返回 `404 Not Found` 或 `403 Forbidden`，前端渲染占位图并提示“无权访问该病历”）。
  - 测试用药打卡逻辑：在每日规定的晨、午、晚三个时间段进行重复确认打卡的拦截（防御预期：联合唯一性约束生效，拒绝冗余生成 `MedicationLog`）。
- **医生端用例**：
  - 测试医生通过Web后台批量浏览、通过、修订和全案驳回病历的功能闭环。
  - **异常模拟**：医生对同一个已经处于 `approved` 状态的病例试图进行二次覆盖或逆向回滚审批节点（测试防御预期：前台隐藏操作按钮组合，若直接调用API接口，后端序列化器抛出状态固化异常 `400 Bad Request`）。

### 8.1.2 YOLO目标切割与大语言模型响应测试
作为系统最具差异化属性的模块，AI检测分析阶段存在因外部网络环境与图片噪音带来的极高不可控性。为此专门针对AI流水线注入了故障注入（Fault Injection）测试。
- **图像失真测试**：上传极端光线（过度曝光/完全逆光）下的全脸照片、非人脸风景图，测试前端预压缩及后端YOLO服务的过滤底线（预期：流水线及时阻断，在轮询查询里抛出 `error_message` 提示“未检测到清晰舌体”，而不是静默等待超时崩毁）。
- **NLP推演容错测试**：针对长文本主诉，使用长文本垃圾语料或不相关的问诊词汇测试大语言模型接口。验证当 JSON 结构化抽取字段丢失（如 `symptoms` 数组为空）时，下游辨证推理引擎的熔断表现（预期：使用基于词典规则的 Fallback 后备降级，防止页面出现白屏 `TypeError: undefined is not an object`）。

## 8.2 性能与并发负载测试

除了保障业务运转无误，系统的鲁棒性是支撑生产级发布的底气。考虑到 Django 与 Celery 调度的实际物理瓶颈，我们采用了本地压测工具模拟高频网络波动。

### 8.2.1 数据库防连接撕裂与长连接测算
我们在开发测试后期遭遇到因服务端长时间静默挂起引起的 MySQL 守护进程断开错误（`OperationalError 2006`）。为复现此情形，我们在客户端建立基于 SSE 的双向聊天流订阅后，掐断所有数据传输10小时并恢复动作。
**修复与重测反馈**：调整 Django 数据库底层的 `CONN_MAX_AGE=28800`，使之与 MySQL 设置相吻合，复测后该连接池失效报错被根除，且在挂起1万个并发假负载会话时，内存表现出平缓上涨但未触及 OOM（Out Of Memory）红线。

### 8.2.2 高峰期异步阻塞压力测试
模拟在“名老中医会诊日”等虚拟极端场景下的流量高峰峰值。利用自动化脚本在 1 分钟内并发注入 200 条含有 5MB 以上原始图片的诊断生成队列。
- **压测表现**：通过后台观察 Celery Worker 的日志吞吐频率。200个 `queued` 的耗时分析任务严格遵循队列纪律（FIFO机制），未发生内存堆溢出导致的主进程阻塞死亡。并且在长达 15 分钟的队列排队积压期间，患者小程序端的查询接口始终返回 `200 OK` 保持通信存活，保证界面的渲染进度更新处于持续动态可视的状态。各项指标表明总体架构成功地将阻塞任务有效解耦。

## 8.3 测试总结论

经过上述三大部分多维度的综合测试与性能调优修正：
基于YOLO与大语言模型的中医智能辅助诊疗项目的前端页面与异步状态轮询能够流畅衔接；后端的并发控制处理及 Celery 推演流水排版稳定，鉴权拦截严密；系统的总体设计与需求分析完全吻合。在实验室及日常演示环境下，平台不仅达成了各项智能辅助辨证指标预期，还交出了良好的错误容忍与自维护答卷。具备在真实环境中推行小型试点研究并进行数据反哺迭代的应用能力。
