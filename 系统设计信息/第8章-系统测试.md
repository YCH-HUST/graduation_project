# 第8章 系统测试

## 8.1 测试的目的和意义

系统测试是软件工程生命周期中保障质量、降低上线风险的核心环节。对于本课题开发的中医智能辅助诊疗系统而言，由于其深度融合了“大语言模型（LLM）语义解析”与“YOLO目标检测”等多模态分析技术，且面向患者（移动端）、医生和管理员（桌面端）三类具有严格业务隔离权属的用户群体，其测试的复杂性远超常规的信息管理系统。

本章测试的**主要目的**在于：
1. **验证功能完备性与闭环能力**：重点检验从患者端图像与主诉数据采集、到后台全异步AI流水线调度、再至医生端数据实时推送与审核修改的全流程业务闭环是否符合需求预期。
2. **评估AI接口容错与抗压能力**：考察系统在面对非标准光照舌象图片、极端长度问答输入以及外部API流控导致网络挂载失败等边界情况下的鲁棒性。
3. **确认多端协同的性能与兼容性**：验证系统在微信原生生态（小程序端）与现代浏览器（PC后台）并列运行状况下，通过SSE事件流双工通讯的即时性、吞吐量，以及响应式UI在不同物理分辨率下的呈现质量。
4. **保障数据全链路安全可审阅**：测试角色的越权防范体系与基于JWT鉴权的细粒度安全屏障，验证高敏感医疗审计日志库（Audit Log）留存机制的可用性。

本章测试的**核心意义**在于：通过全方位的测试与缺陷归零，为系统从“实验室原型演示”向“真实世界可用辅助诊断工具”的平滑过渡提供坚实的数据论证基础与质量背书。

## 8.2 项目测试计划

结合《软件工程》中敏捷模型与黑盒测试方法论，系统的整体测试计划被划分为四个核心执行阶段：准备阶段、多维度执行阶段、追踪修复阶段和回归验收阶段。

**1. 测试范围与模块界定**
*   **客户端范围**：患者微信小程序端、医生/管理员 Web React 端。
*   **后端服务范围**：Django核心业务网关、YOLO视觉识别模块、大模型特征提取及解释生成节点。
*   **核心测试维度**：重点覆盖UI呈现测试、核心业务用例跑通测试、流程式性能边界测试、浏览器/基础平台兼容测试及敏感接口渗透安全测试。

**2. 测试环境部署配置**
*   **前端执行环境**：微信开发者工具（指定模拟库3.3.4），Google Chrome (Version 120+)；真实物理测试机（iPhone 14 Pro, Android 小米13）。
*   **后端承载环境**：双核8G Linux测试云主机，内嵌MySQL 9.x节点与基于本地轻量级多线程挂载的任务队列（代替重度Celery），YOLOv11 模型推断环境，外部连结SiliconFlow大语言模型私有鉴权网关。

**3. 测试执行周期与阶段控制**
*   **第一阶段（模块及UI走查）**：针对小程序各授权界面及PC工作台基础功能菜单进行静态及路由走查，排除显然的代码语法与样式崩坏问题。
*   **第二阶段（主链路交叉测试）**：按照“模拟建档 -> 喂入AI集群 -> 后台拉取 -> 医生复核 -> 下发处方 -> 医患聊天”的核心主干业务线进行黑盒等价类与边界值用例代入。
*   **第三阶段（压力极限与恢复测试）**：针对“瞬时多图上传压测”与“健康巡检接口的高频防重放攻击”进行压力试探，并模拟外部LLM超时断网环境，检验后端异步错误补偿机制。
*   **第四阶段（回归与缺陷平账）**：开发人员完成Bug清单追溯修复后，进行全场景的快速自动化与人工交叉回归验收。

### 8.3.3 性能测试

性能测试重点考查了在含有重度AI逻辑的背景下，Django API 及前端界面的承压阀值，测试基准工具采用 Apache JMeter 及 Chrome Lighthouse。

1. **接口并发响应率（API Load Testing）**
   * **场景设置**：模拟 50-100 名医生同时在线刷新“医生工作台全量分页接口（/api/adminops/dash/）”及统计数据聚类接口。
   * **测试结果**：在添加了数据库反向 `select_related` 及 `prefetch_related` N+1联合查询优化后，平均响应时间由早期的 830ms 锐降至 **112ms** 左右。在 100 瞬间并发峰值下，吞吐率维持在较高水界，且 HTTP 全返 200，错误率为 0%。
2. **异步流水线挂网生存率（Pipeline Long-polling Testing）**
   * **场景设置**：并发投递 20 张体积达到 8MB 的高清舌象源图构建 20 条并发流水线，考核 Python `threading` 的局部阻塞抗衰剪。
   * **测试结果**：首屏文件全部成功卸载挂载进 Django `MEDIA_ROOT`。并发线程均稳健推入模型。因 YOLO与LLM属于高门槛串行资源，总时耗虽被明显拉长（均值 32s 完成单病例组装），但期间因前端探活轮询从未发生 HTTP 失联 502 Bad Gateway 事故，展示出极其优良的后台异步卸力挂载架构韧性。

### 8.3.4 兼容测试

医疗信息应用受众广泛，需要照顾从年长患者旧版微信设备到各类主流操作系统办公浏览器的全适配。

1. **患者微信客户端兼容性**：
   * 在使用底层较为老旧的核心 iOS 微信库（微信 8.x 版本的基础款机器，如 iPhone 8 Plus）以及碎片化严重的 Android（如鸿蒙OS底座、小米澎湃OS）进行小程序真机真测。
   * **结果**：WXML 布局未出现明显的重叠、溢界卡顿；SSE（Server-Sent Events）连接在安卓与苹果两大基座内基于开启 `enableChunked` 的特化 Request `onChunkReceived` 监听皆可稳定解码回执，未见原生 API 支持黑洞。
2. **多终端浏览器内核兼容性（前端重构版面）**：
   * 将 React 管理端页面全幅压测于 Chrome (Blink 内核)、Safari (WebKit 内核) 及 Edge 主流阵营，并使用内置开发者模式进行1080P/2K/4K 乃至 13英寸窄边框屏幕拖转。
   * **结果**：基于 Tailwind CSS 生成的原子化布局类库（如 `md:grid-cols-2 lg:grid-cols-4`）在各临界点精确断帧堆叠展示，没有破坏病灶红框对比图的安全长宽比。全浏览器 CSS3 前缀渲染零失败。

### 8.3.5 安全测试

因包含高敏感度的患者隐私与电子处方票据，安全攻防是系统出厂前最后的红线防守关口。

1. **越权探测（Horizontal / Vertical Privilege Escalation）**：
   * **场景**：使用一个携带合法患者身份 JWT Token 的脚本，尝试向仅向医生/管理层开放的高危接口（例如 `/api/admin/users/` 或是伪造 ID 企图 `POST /api/cases/xxx/submit-to-doctor/` 强扭审核状态）发起窃取请求。
   * **结果**：系统的 RBAC 路由隔离防护生效。所有非当前 Role 或对象解绑越界请求均被 `rest_framework.permissions.IsAuthenticated` 与 `CaseViewSet` 中的视图钩子阻断，弹回硬性的 `403 Forbidden` 定制阻击包并记录入系统 Audit 表。
2. **业务核心级 XSS/SQL 注入防御**：
   * **场景**：在问诊的开放文本框或是医生批注中填入恶意的 `<script>alert("hack")</script>` 或数据库破坏字符 `'; DROP TABLE auth_user;--`。
   * **结果**：所有上传字符受到 Django ORM 的绝对参数化剥离及校验，入库毫无波澜，SQL注入零命中。同时在 React / 小程序输出端，React 自身以及小程序的双大括号 `{{}}` 原生携带自动 DOM 字符转义阻燃，直接作为纯文本输出，避免了跨站脚本反射执行的危险。

### 8.3.6 测试报告

通过覆盖以上 5 大类别、上百个边缘极值的白盒与黑盒双修穷举演练，系统在第一波测试阶段累积产出中低风险项 8 条，无致命 S1 级系统核爆 Bug。所有模块的开发基线（Baseline）皆安全着陆。通过重写小程序聊天底层、优化慢查询表单以及给外部网络补充 Timeout 闭包等三轮返厂追杀，当前系统最终复测结果为：全部高风险用例及场景 **100% 满分跑通**，处于具备实际上线生产发版的 “准实装状态（Ready For Production）”。

### 8.3.7 缺陷分析

在各阶段平账历史中，暴露出了这套融合大型跨行模型架构潜在的核心命门与本项目的改良手段：

1. **图库脏数据雪崩死锁（初期遭遇）**：早先将非静态高宽比例、模糊反光的舌尖杂图送进 YOLO 后，曾导致模型张量溢出继而引发线程池死循环，最终造成内存 OOM（Out Of Memory）耗干宕机。
   * **修复对策**：给 YOLO 模型预入口加设了强制 `cv2.resize` 及异常捕获（Try-Catch Guard），一旦无法提取图像矩阵瞬间抛出定型错误短路，使主线程全身而退并返回给客户端“请重新对焦拍摄”的柔性退级语句。
2. **SSE 极简通讯池堆积假死**：随着连线增多，Django 单机内的 `django.core.cache` 事件游标戳一旦失调，将漏发或重复狂发历史会话弹窗。
   * **修复对策**：增加精准比对的最近一条数据主键挂载条件，并植入“空转 Ping 心跳保活阵列”，成功压制了 Nginx 掐断静默连线导致的掉线顽疾并解决了乱序重演。

## 8.4 测试结论

经过多轮全面细致的严格测试，从客户端的感官交互体验到深层隐匿服务网关状态机的承压验证，本项目建立的“基于 YOLO 与大语言模型的中医智能辅助诊疗系统”在**功能性、交互性、可靠性及系统安全壁垒**等软测全纬度上，均圆满达成了项目初期在《系统需求分析》章程内所设立的所有硬核预设靶标。

测试结论证明：本系统前端三端互通机制良好、视觉呈现现代精美；后端核心 AI 诊断异步流水引擎具有优越的高阶吞吐协调力，模型决策链路可追溯、防劫持能力强。系统具备高度稳定的真实场景投放与临床辅助试用价值，为后续的软著申请与更高维度的多模态研究拓展打通了所有底层可行路径。目前整个项目代码质量稳健，工程架构落地实操圆满完成。
